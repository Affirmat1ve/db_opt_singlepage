УРОВЕНЬ 0: CLICKHOUSE — БАЗОВЫЕ КОНЦЕПЦИИ

1. ТЕОРЕТИЧЕСКИЙ БЛОК

1.1 Что такое ClickHouse

ClickHouse — это система управления базами данных, созданная в Яндексе в 2016 году для сервиса Яндекс.Метрика. Её главная задача — мгновенно отвечать на аналитические вопросы по огромным объёмам данных: миллиардам строк логов, событий, транзакций.

Представьте обычную таблицу в базе данных. В традиционных системах данные хранятся строками: когда вы читаете запись, система загружает все поля подряд — id, имя, email, дату. Это удобно, когда нужно получить полную информацию об одном пользователе.

ClickHouse хранит данные колонками. Каждая колонка — отдельно. Если вам нужно посчитать средний возраст пользователей, система прочитает только колонку age, игнорируя все остальные данные. Это даёт огромный выигрыш в скорости при аналитических запросах.

1.2 Как ClickHouse работает с данными

Когда вы вставляете данные в ClickHouse, они не перезаписывают старые записи. Новые данные добавляются как отдельные части (parts). Позже, в фоновом режиме, система автоматически объединяет эти части — процесс называется merge. Это нужно для оптимизации: отсортировать данные, построить индексы, сжать информацию.

Важный момент: эти фоновые процессы идут постоянно, даже когда вы не выполняете запросы. ClickHouse всегда что-то делает: сливает части, очищает старое, обновляет статистику. Это нормально и заложено в архитектуру.

1.3 Многопоточность и ресурсы

ClickHouse создан для использования всех доступных ресурсов сервера. Один аналитический запрос может задействовать несколько ядер процессора одновременно. Система сама распараллеливает вычисления, чтобы получить результат как можно быстрее.

Также ClickHouse активно использует оперативную память для кэширования. Чем больше данных помещается в память, тем быстрее работают запросы. Система «жадная» до свободной RAM — это особенность, а не ошибка.

1.4 Для каких задач подходит ClickHouse

✅ Отлично подходит:

  • Веб-аналитика и счётчики посещений
  • Мониторинг серверов и приложений
  • Финансовые отчёты и биржевые данные
  • Телеметрия IoT-устройств
  • Любые задачи, где данные в основном добавляются, а не меняются

❌ Не подходит:

  • Интернет-магазины с частыми обновлениями остатков
  • Банковские транзакции с частыми изменениями записей
  • Приложения, где важны сложные UPDATE и DELETE

Причина проста: ClickHouse оптимизирован для быстрой вставки и чтения, а не для частых изменений. Операции обновления выполняются, но медленно и с оговорками.

1.5 Минимум для понимания производительности

Чтобы дальше разбираться с наблюдением за работой системы, запомните три ключевых момента:

Первое: Фоновые процессы — это норма. Если вы видите активность со стороны ClickHouse, когда «ничего не делаете» — система просто оптимизирует данные.

Второе: Многопоточность — это особенность. Запрос может использовать несколько ядер процессора одновременно. Это не ошибка, а способ ускорить получение результата.

Третье: Память — критический ресурс. ClickHouse использует RAM для кэширования. Если памяти не хватает и система начинает использовать диск вместо памяти, производительность падает резко и значительно.

2. КОНТРОЛЬ ЗНАНИЙ

Вопрос 0.1

Вопрос: Почему ClickHouse хранит данные колонками, а не строками?

Варианты ответов:

  1. Это экономит место на диске
  2. Это позволяет быстрее выполнять аналитические запросы, читая только нужные поля
  3. Это упрощает резервное копирование
  4. Это требование стандарта SQL

Правильный ответ: 2

Объяснение: Колоночное хранение позволяет при запросе читать только те колонки, которые действительно нужны. Например, при запросе SELECT AVG(age) FROM users система прочитает только колонку age, игнорируя все остальные данные. Это даёт огромный выигрыш в скорости при работе с большими объёмами информации.

Вопрос 0.2

Вопрос: Что такое процесс MERGE в ClickHouse и когда он происходит?

Варианты ответов:

  1. Это ручная операция объединения таблиц, которую запускает администратор
  2. Это автоматическое фоновое слияние частей данных для оптимизации хранения и чтения
  3. Это процесс резервного копирования базы данных
  4. Это обновление версии ClickHouse

Правильный ответ: 2

Объяснение: Когда данные вставляются в ClickHouse, они добавляются как отдельные части. Позже система автоматически объединяет (сливает) эти части: сортирует данные, строит индексы, применяет сжатие. Этот процесс называется MERGE и происходит в фоновом режиме постоянно. Именно поэтому ClickHouse проявляет активность, даже когда пользователь не выполняет запросы.

Вопрос 0.3

Вопрос: Почему ClickHouse не рекомендуется для систем с частыми операциями UPDATE и DELETE?

Варианты ответов:

  1. Эти операции полностью запрещены в ClickHouse
  2. ClickHouse оптимизирован для вставки и чтения, а изменение данных выполняется медленно и неэффективно
  3. Обновления могут повредить данные
  4. Это нарушает лицензионное соглашение

Правильный ответ: 2

Объяснение: Архитектура ClickHouse заточена под сценарии, где данные в основном добавляются, а не меняются. Операции UPDATE и DELETE технически возможны, но выполняются через перезапись частей данных, что требует значительных ресурсов и времени. Для транзакционных систем с частыми изменениями лучше подходят традиционные СУБД вроде PostgreSQL или MySQL.

3. ЧЕКЛИСТ ГОТОВНОСТИ

Перед переходом к следующему уровню убедитесь, что вы понимаете:

  • ClickHouse — колоночная аналитическая СУБД, а не транзакционная
  • Данные хранятся отдельно по колонкам для ускорения аналитических запросов
  • Фоновые процессы MERGE — нормальная часть работы системы
  • ClickHouse использует многопоточность для ускорения запросов
  • Оперативная память критически важна для производительности
  • ClickHouse подходит для задач с преимущественной вставкой данных, а не их изменением

Если все пункты понятны — вы готовы двигаться дальше!

УРОВЕНЬ 1: СРЕДСТВА МОНИТОРИНГА ОПЕРАЦИОННОЙ СИСТЕМЫ

1.1 Описание категории инструментов

Средства мониторинга операционной системы — это базовый уровень наблюдения за работой ClickHouse. Поскольку ClickHouse работает на Linux и интенсивно использует системные ресурсы (CPU, память, дисковый ввод-вывод), мониторинг на уровне ОС критически важен для:

  • Выявления узких мест производительности
  • Понимания потребления ресурсов процессом clickhouse-server
  • Диагностики проблем с оборудованием
  • Планирования масштабирования

1.2 Основные инструменты

Инструмент Назначение Ключевые метрики
top/htop Мониторинг процессов и CPU %CPU, %MEM, RES, TIME+
vmstat Виртуальная память, CPU r, b, swpd, free, bi, bo, wa
iostat Дисковый I/O r/s, w/s, rkB/s, wkB/s, %util, await
iotop I/O по процессам DISK READ, DISK WRITE, IO%

Шаг 1: Скачивание

curl https://clickhouse.com/ | sh

Шаг 2: Установка как сервис

sudo ./clickhouse install

sudo clickhouse start

clickhouse-client

Шаг 5: Теперь можно писать запросы напрямую после :)

SELECT 1

Теперь работа будет устроена следующим образом: оставьте это окно, откройте новое. Для нового окна: 1) Запустите указанную утилиту. Посмотрите на показатели при фоновом процессе. 2) Вернитесь к окну 1. с открытым clickhouse-client и запустите в нем запрос:

SELECT
count(),
avg(number),
max(number)
FROM numbers(10000000);

3) В этот же момент смотрите на окно 2. со статистиками и то, как они меняются.

Вопрос 1

Ситуация: Вы запустили утилиту vmstat для мониторинга системы во время выполнения аналитического запроса в ClickHouse. Обратили внимание, что значение в колонке wa (I/O wait) стабильно держится на уровне 45%.

Вопрос: Что означает это значение и какие действия необходимо предпринять?

Варианты ответов:

  1. Это нормальное значение, ClickHouse использует 45% CPU для вычислений.
  2. 45% времени процессор простаивает в ожидании операций ввода-вывода с диском — это критическое узкое место.
  3. 45% оперативной памяти используется под кэш файловой системы.
  4. Сеть работает медленно, 45% пакетов теряется при передаче данных.

Правильный ответ: 2️⃣ 45% времени процессор простаивает в ожидании операций ввода-вывода с диском — это критическое узкое место.

Объяснение: Колонка wa (iowait) показывает процент времени, когда CPU не занят полезной работой, а ждет завершения операций чтения/записи диска. Значение 45% означает, что почти половина времени процессора тратится впустую. Для SSD нормальным считается значение <5%, для HDD <10%. Рекомендуемые действия: проверить утилизацию диска через iostat, рассмотреть замену HDD на SSD/NVMe, оптимизировать запросы для снижения нагрузки на диск.

Вопрос 2

Ситуация: Вы смотрите на процесс clickhouse-server в утилите top. В колонке %CPU отображается значение 385.2%.

Вопрос: Сколько ядер процессора фактически использует ClickHouse в этот момент?

Варианты ответов:

  1. Примерно 3.85 ядра (так как 100% = одно ядро).
  2. 385 ядер одновременно.
  3. 38.5% от одного ядра.
  4. Это ошибка измерения, значение не может превышать 100%.

Правильный ответ: 1️⃣ Примерно 3.85 ядра (так как 100% = одно ядро).

Объяснение: В Linux утилита top показывает загрузку CPU в процентах от одного логического ядра. Значение 100% означает полную загрузку одного ядра. ClickHouse — многопоточная СУБД, которая может параллельно использовать несколько ядер для выполнения одного запроса. Поэтому 385.2% означает, что полностью загружены 3 ядра и ещё одно загружено на 85%. Если на сервере 8 ядер, то используется примерно 48% вычислительной мощности.

Вопрос 3

Ситуация: Вы проверили статистику дисковой подсистемы через iostat -x. Для диска, на котором хранятся данные ClickHouse, вы увидели следующие показатели: утилизация диска (%util) составляет 98.9%, а среднее время отклика (await) — 45.67 мс.

Вопрос: Какой вывод можно сделать о состоянии дисковой подсистемы?

Варианты ответов:

  1. Диск работает в штатном режиме, нагрузка оптимальна.
  2. Диск перегружен, время отклика слишком высокое для современной СУБД.
  3. Необходимо увеличить количество операций чтения.
  4. Необходимо отключить кэширование записи.

Правильный ответ: 2️⃣ Диск перегружен, время отклика слишком высокое для современной СУБД.

Объяснение: Утилизация 98.9% означает, что диск практически постоянно занят обработкой запросов (очередь не успевает обрабатываться). Время отклика 45.67 мс является очень высоким показателем. Для NVMe SSD норма — менее 1 мс, для SATA SSD — менее 5 мс, для HDD — менее 10 мс. Такое значение указывает на то, что диск не справляется с нагрузкой, что напрямую тормозит работу ClickHouse. Рекомендуется перенести данные на быстрые NVMe накопители.
