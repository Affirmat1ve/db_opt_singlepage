УРОВЕНЬ 0: CLICKHOUSE — БАЗОВЫЕ КОНЦЕПЦИИ

1. ТЕОРЕТИЧЕСКИЙ БЛОК

1.1 Что такое ClickHouse

ClickHouse — это система управления базами данных, созданная в Яндексе в 2016 году для сервиса Яндекс.Метрика. Её главная задача — мгновенно отвечать на аналитические вопросы по огромным объёмам данных: миллиардам строк логов, событий, транзакций.

Представьте обычную таблицу в базе данных. В традиционных системах данные хранятся строками: когда вы читаете запись, система загружает все поля подряд — id, имя, email, дату. Это удобно, когда нужно получить полную информацию об одном пользователе.

ClickHouse хранит данные колонками. Каждая колонка — отдельно. Если вам нужно посчитать средний возраст пользователей, система прочитает только колонку age, игнорируя все остальные данные. Это даёт огромный выигрыш в скорости при аналитических запросах.

1.2 Как ClickHouse работает с данными

Когда вы вставляете данные в ClickHouse, они не перезаписывают старые записи. Новые данные добавляются как отдельные части (parts). Позже, в фоновом режиме, система автоматически объединяет эти части — процесс называется merge. Это нужно для оптимизации: отсортировать данные, построить индексы, сжать информацию.

Важный момент: эти фоновые процессы идут постоянно, даже когда вы не выполняете запросы. ClickHouse всегда что-то делает: сливает части, очищает старое, обновляет статистику. Это нормально и заложено в архитектуру.

1.3 Многопоточность и ресурсы

ClickHouse создан для использования всех доступных ресурсов сервера. Один аналитический запрос может задействовать несколько ядер процессора одновременно. Система сама распараллеливает вычисления, чтобы получить результат как можно быстрее.

Также ClickHouse активно использует оперативную память для кэширования. Чем больше данных помещается в память, тем быстрее работают запросы. Система «жадная» до свободной RAM — это особенность, а не ошибка.

1.4 Для каких задач подходит ClickHouse

✅ Отлично подходит:

  • Веб-аналитика и счётчики посещений
  • Мониторинг серверов и приложений
  • Финансовые отчёты и биржевые данные
  • Телеметрия IoT-устройств
  • Любые задачи, где данные в основном добавляются, а не меняются

❌ Не подходит:

  • Интернет-магазины с частыми обновлениями остатков
  • Банковские транзакции с частыми изменениями записей
  • Приложения, где важны сложные UPDATE и DELETE

Причина проста: ClickHouse оптимизирован для быстрой вставки и чтения, а не для частых изменений. Операции обновления выполняются, но медленно и с оговорками.

1.5 Минимум для понимания производительности

Чтобы дальше разбираться с наблюдением за работой системы, запомните три ключевых момента:

Первое: Фоновые процессы — это норма. Если вы видите активность со стороны ClickHouse, когда «ничего не делаете» — система просто оптимизирует данные.

Второе: Многопоточность — это особенность. Запрос может использовать несколько ядер процессора одновременно. Это не ошибка, а способ ускорить получение результата.

Третье: Память — критический ресурс. ClickHouse использует RAM для кэширования. Если памяти не хватает и система начинает использовать диск вместо памяти, производительность падает резко и значительно.

3. ЧЕКЛИСТ ГОТОВНОСТИ

Перед переходом к следующему уровню убедитесь, что вы понимаете:

  • ClickHouse — колоночная аналитическая СУБД, а не транзакционная
  • Данные хранятся отдельно по колонкам для ускорения аналитических запросов
  • Фоновые процессы MERGE — нормальная часть работы системы
  • ClickHouse использует многопоточность для ускорения запросов
  • Оперативная память критически важна для производительности
  • ClickHouse подходит для задач с преимущественной вставкой данных, а не их изменением

Если все пункты понятны — вы готовы двигаться дальше!
